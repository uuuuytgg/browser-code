## Current UX / Product Reality

The sidepanel is currently not a conversational agent UI.

It is still a minimal action launcher:

- capture page context
- send a task to the local bridge
- show very basic status text

This is why the user saw:

- `Status: processing`
- `Task ... is processing.`

These mean:

- the task POST was accepted by the bridge
- the extension did not yet show the final task result

## Important User Feedback To Preserve

The user wants the product direction to become:

- a natural language chat interface in the browser sidepanel
- agent answers shown as messages
- knowledge-base querying available through conversation
- current task actions moved into quick buttons near the input bar

Desired quick actions:

- save current page
- summarize current video
- scan page resources
- save selected text
- search knowledge base

This should replace the current plain button list layout.

## English Copy Meaning

The user explicitly asked what the English strings mean.

Current strings in the sidepanel:

- `Capture the current page and send a controlled task to the local bridge.`
  - Means: capture the current web page, then send the task to the local backend service.
- `Status: processing`
  - Means: task is still running.
- `Task xxx is processing.`
  - Means: backend accepted the task but has not returned the final result to the UI.
- `Ready to capture the current page.`
  - Means: ready state before clicking a button.
- `Capturing page context...`
  - Means: extracting current page info from the active tab.
- `Sending task to local bridge...`
  - Means: sending the task to the local backend agent.

## Known Problems

### 1. Chinese labels are garbled

File:

The action labels are currently mojibake, for example:

- `淇濆瓨褰撳墠缃戦〉`

This should be replaced with proper Chinese labels:

- 总结当前视频
- 扫描页面资源
- 保存选中文本
- 搜索知识库

### 2. Extension never polls final task status

Files:

- `apps/extension/src/sidepanel/AppStage4.tsx`
- `apps/extension/src/bridge-client/localhost-client.ts`
- `apps/local-bridge/src/localhost-server.ts`

Current behavior:

- extension only `POST /tasks`
- bridge returns immediate `{ status: "processing" }`
- extension shows that processing state
- extension never follows up with `GET /tasks/:taskId`

Result:

- user sees only "processing"
- even if backend eventually succeeds, sidepanel does not show final result

This is the main reason the user is confused about whether actions actually completed.

### 3. Runtime vault location depends on launch directory

Observed from `browser-code doctor` in the user's terminal:

- `vaultDir: D:\\ClaudeData\\vault`
- `tempDir: D:\\ClaudeData\\temp`

This happened because the user launched `browser-code` from `D:\ClaudeData`.

Important implication:

- backend writes do **not** necessarily go to repo-local `D:\ClaudeData\browser agent\vault`
- they go to the runtime working directory vault unless env vars override it

### 6. Transcript fallback is still placeholder-grade

Files:

- `packages/tool-video/src/transcript-fetcher.ts`
- `packages/tool-video/src/bilibili-transcript.ts`
- `packages/tool-video/src/ffmpeg-adapter.ts`

Current behavior:

- transcript extraction relies on captured page HTML
- if no transcript is found, it returns `need_audio_transcription`
- the ffmpeg extraction path is still placeholder-only and does not perform real transcription

So summarize-video currently has limited success conditions.

### 7. Runtime is still on mock provider

Observed from `browser-code doctor` in the user's terminal:

- `modelProvider: "mock"`

Implications:

- no real LLM provider is currently configured
- no API key entry flow exists in the product yet
- the backend is not doing true natural-language agent reasoning with a production model

Important nuance:

- tool execution itself is not fully blocked by the lack of a real model
- the current local bridge uses a deterministic mock orchestration path in `apps/local-bridge/src/runtime-client.ts`
- that means some tool chains can still run, but the quality ceiling is low and behavior is rigid

### 8. Different buttons currently collapse onto the same source URL dedupe path

Files:

- `packages/tool-vault/src/save-note.ts`
- `packages/tool-vault/src/dedupe.ts` if further inspection is needed

Current behavior:

- note dedupe is keyed by `source_url`
- it does not sufficiently distinguish action intent such as:
  - `save_page`
  - `summarize_video`
  - `save_selection`

User-visible impact:

- clicking different buttons on the same video/page URL can keep returning the same existing note
- the first saved note can effectively "win"
- later actions may appear to do nothing or appear to overwrite semantics without creating a new file

This matches the user's observed behavior that repeated clicks across different actions still led to one article file.

## Recommended Next Steps

### Priority 1: Fix sidepanel product shape

Convert sidepanel from action launcher into chat-first UI:

- message list
- user input box
- send button
- quick action chips/buttons above the input
- final results rendered as conversation messages

### Priority 2: Remove English and fix Chinese encoding

Replace current sidepanel copy with proper Chinese.

### Priority 3: Add task polling

Implement:

- `POST /tasks`
- then poll `GET /tasks/:taskId`
- show final `done`, `error`, or `need_confirmation`

Without this, the current UX will keep looking broken.

### Priority 4: Make runtime paths explicit

Add explicit env/config handling so vault location is not surprising.

Suggested:

- show current vault path in UI
- let user configure it
- or force repo-local defaults during dev

### Priority 5: Verify summarize_video end-to-end

Test on a real Bilibili page:

1. capture page context
2. extract transcript
3. save note to `vault/videos`
4. confirm sidepanel shows final result

If transcript extraction fails:

- either implement real audio transcription
- or surface a clear "not supported yet" message instead of silent processing

### Priority 6: Fix dedupe semantics

Do not dedupe different capture intents purely by `source_url`.

At minimum, dedupe should distinguish:

- action type
- content type
- source URL

Otherwise `save_page`, `summarize_video`, and `save_selection` for the same page can collapse into one note.
